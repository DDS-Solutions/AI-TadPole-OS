This document defines the core concepts and terminology used within the Tadpole OS ecosystem. This glossary is designed to provide consistency for both human operators and AI assistants.

## Core Concepts

### Debounced Persistence
An architectural pattern where high-frequency database writes (e.g., token usage) are buffered in memory and flushed to disk at regular intervals (e.g., every 10 seconds) to reduce contention and improve performance.

### Deep Hierarchical Synthesis
The shared memory space for a swarm mission. All findings, synthesized reports, and system logs within a mission are broadcast to the Context Bus, allowing agents to "hear" each other's progress without direct peer-to-peer communication.

### Neural Sector Fault (NSF)
A failure in a specific layer of the engine (e.g., Core, Provider, or UI). NSFs are typically logged with trace details to help developers isolate the sector (Backend vs Frontend vs LLM) responsible for the error.

### Neural Oversight
The governance layer of Tadpole OS responsible for cluster-wide strategic optimization. It proactively identifies organizational inefficiencies and proffers configuration changes to the human operator.

### Optimization Proposal
A structured recommendation (`SwarmProposal`) generated by the engine to improve a mission cluster's efficiency. Proposals may include model upgrades, role reassignments, or skill extensions.

### Strategic Command
The interactive interface within the Hierarchy Layer (Alpha Node) where operators review, authorize, or dismiss Neural Oversight proposals.

### Neural Handoff (Voice-to-Swarm)
The process of capturing human voice intent, transcribing it via high-fidelity engines (Groq Whisper), and delivering the strategic text to the Agent of Nine for autonomous swarm dispatch.

### Neural Synthesis (Audio Feedback)
The process of generating high-fidelity vocal responses from agents using the server-side architecture (OpenAI TTS). This enables a true hybrid interaction where the agent "speaks" back to the user during neural handshakes.

### Deep Hierarchical Synthesis
The convergence phase where an Alpha Node merges findings from depth-2+ sub-agents. Missions involving deep hierarchical synthesis are typically those with swarm depth > 2 and require the Alpha to reconcile findings from multiple specialist domains.

### Swarm Depth
The recursive level of sub-agent spawning. Tadpole OS enforces a strict **limit of 5** (configured in `AppState`) to prevent infinite recursion.
- **Depth 0**: Direct User-to-Agent task.
- **Depth 1**: Overlord spawned an Alpha Node.
- **Depth 2-5**: Secondary and tertiary specialists.

### Parity Guard
An automated infrastructure service that verifies 100% synchronization between the codebase (Rust routes) and multi-layered documentation (OpenAPI, API Reference). It acts as a gatekeeper to prevent documentation drift.

### Forward-Only Parity Gate
A development strategy where features are considered incomplete—and the build is blocked—if they lack synchronous documentation. This ensures that parity is preserved by moving only in a "forward-synced" state.

### Non-Repudiation
A security property ensuring that an agent action or system decision cannot be denied by the entity that performed it. In Tadpole OS, this is achieved through Ed25519-signed Merkle audit entries linked to specific `AUDIT_PRIVATE_KEY` identities.

### Merkle Hub
The central backend service responsible for managing the tamper-evident audit trail, cryptographic signatures, and non-repudiation verification.

### Parallel Swarming (PERF-06)
The engine's ability to execute multiple tool calls (e.g., spawning 3 researchers) concurrently. This uses Rust's `FuturesUnordered` to significantly reduce latency in hierarchical swarms compared to sequential tool execution.

### Swarm Lineage (Recruitment Chain)
The deterministic tracking of agent ancestry within a mission. The engine uses this lineage to block any agent from recruiting an ancestor, preventing circular "ping-pong" loops (e.g., A -> B -> A).

### Unified Synthesis Engine
The backend logic in `runner.rs` that standardizes how agents from different providers (Gemini, Groq) merge their findings into a single, structured report.

### Self-Healing Retry (Groq)
A resilience layer in the Groq provider that automatically detects `tool_use_failed` errors (400) caused by malformed JSON. It performs an immediate, corrective retry to ensure mission continuity without manual intervention.

### Groq Whisper
A high-fidelity speech-to-text model (`whisper-large-v3`) utilized by the backend to ensure precise transcription of user intent for the sovereignty handoff.

### Vector Neural Memory
The persistent integration of **LanceDB** and **Apache Arrow** directly into the engine's data layer. It provides agents with long-term semantic recall of previous finding abstractions without impacting their active context limits.

### Split-Brain Architecture
Tadpole OS's dual-database governance. Traditional **SQLite** handles deterministic data (mission logs, financial budgets, UI hierarchy), while **LanceDB** exclusively manages high-dimensional embeddings for associative agent recall.

### RAG Scope
A temporary, highly localized sandboxed LanceDB directory (`scope.lance`) spun up when a mission cluster activates. Agents query this focused semantic space natively before making external API requests to prevent hallucinations.

### Continuity Scheduler
A dedicated asynchronous daemon (`agent/continuity`) that evaluates `ScheduledJob` cron expressions every 60s to trigger autonomous AI missions.

### SanitizedJson
A utility wrapper used in serialization to automatically prune long strings (exceeding 2KB) from JSON payloads, preventing database bloat and memory exhaustion.

### Structured Working Memory
A persistent JSON-based scratchpad assigned to each agent, allowing them to maintain "Chain of Thought" reasoning and mission milestones across multiple turns.

### Workflow Engine
A deterministic pipeline manager (`agent/continuity/workflow.rs`) that orchestrates multi-step mission sequences across different agents.

### Budget Guard (Persistent Metering)
The kernel-level financial enforcement module. It tracks USD expenditure in SQLite, ensuring that budgets persist across restarts and are enforced at the network/LLM gate.

### Merkle Audit Trail
A tamper-evident cryptographic ledger where every critical action (ToolCall, ConfigChange) is SHA-256 linked to its predecessor. Any database tampering breaks the "Hash Chain," triggering a security fault.

### Shell Safety Scanner
A proactive defense layer that scans agent-generated scripts for environment variable leaks or sensitive token references before execution.

### Agent Health Monitoring
A real-time telemetry system that tracks failure counts and operational status. Nodes transition through three primary states:
- **Healthy**: `failureCount` is 0.
- **Degraded**: `failureCount` > 0 but below threshold.
- **Throttled**: `failureCount` >= 3. Automatically suspended to protect system integrity.

### Manual Reset
An administrative action triggered via the UI or API (`POST /v1/agents/:id/reset`) that clears an agent's `failureCount`, resets its `lastFailureAt` timestamp, and returns it to an `idle` state.

### Memory Pressure
A critical resource metric representing the percentage of host and process RAM utilization. High memory pressure triggers defensive alerts in the dashboard to prevent OOM (Out Of Memory) failures.

### Sandbox Detection
A proactive investigative primitive that identifies if the engine is running within a container (Docker/K8s) or an isolated virtual environment.

### Model Store
The sovereign interface for managing intelligence assets. It allows operators to browse, pull, and manage local models (via Hugging Face or Ollama) with integrated VRAM hardware matching to prevent OOM failures.

### Privacy Mode (Shield)
A high-visibility "Hard Gate" for mission-critical privacy. When enabled, the engine explicitly blocks all traffic to external cloud providers (OpenAI, Gemini, etc.) and routes all missions to the **Local Swarm**.

### Merkle Integrity Score
A 0-100% security rating derived from a cryptographic audit of the Merkle Chain. A score of 100% ensures that every tool call and decision is authentic and untampered.

### Resource Guard
The real-time telemetry subsystem responsible for hardware-level monitoring (CPU, RAM) and sandbox discovery.

### Sovereign Link
A direct communication channel established in the UI (denoted by a message icon in the `CommandTable`) that instantly focuses the Neural Chat onto a specific agent node.

### HATEOAS (Hypermedia as the Engine of Application State)
A REST Level 3 architectural pattern used in the Tadpole API. It provides discoverable navigation via `_links` in response envelopes, allowing the UI to navigate resources without prior endpoint knowledge.

### Problem Details (RFC 9457)
The standardized machine-readable error format used by the engine. It ensures that API failures provide structured context (Type, Title, Detail) for graceful UI handling.

### Neural Pulse
The high-frequency WebSocket event stream used for real-time telemetry. It broadcasts engine events, agent thoughts, and system metrics to maintain sub-second UI reactivity.

### Neural Engine Access Token
The primary security credential (configured in **System Configuration** and `.env`) used to verify secure WebSocket and API connections between the frontend interface and the core Rust engine. Formerly referred to as the "Neural Token".

### Orphan Sweeper
A background maintenance task that purges stale mission directories (`scope.lance`) from the `workspaces/` folder to prevent unbounded disk growth.

### Deduplication Thresholding
A vector-based safety check using cosine distance (controlled by `LANCEDB_DEDUPE_THRESHOLD`) to prevent redundant insights from being written to an agent's long-term memory.

### Cinematic Depth View
A high-fidelity analysis modal in the UI providing 360-degree observability into a single neural transmission, including lineage paths and model configuration.

### Lineage Stream
A real-time telemetry sidebar visualizing the propagation of instructions through the swarm hierarchy in a scrollable temporal feed.

### Neural Forge
The administrative interface within `ModelManager` where new model nodes are provisioned, configured, and added to the intelligence inventory.

## Operational Terms

### Budget USD (Fiscal Gate)
A hard limit on the monetary cost of a mission. The engine calculates token costs in real-time and will automatically pause the mission if this limit is reached.

### Oversight Gate (Human-in-the-Loop)
The security interception layer. It pauses agent execution during "sensitive" tool calls (like writing files or completing missions) until a human user approves the action via the dashboard. It also enforces the **Requires Oversight** flag on individual agents, ensuring all their actions are gated.

### Requires Oversight
An agent-level governance flag. When enabled, every tool call initiated by the agent is automatically intercepted by the Oversight Gate, regardless of the tool's individual safety rating. This is used for "Junior Agents" or high-risk autonomous nodes.

### Mission-Level Quotas
Granular fiscal limits applied to a specific Mission Cluster. These complement the global and per-agent budgets, allowing for precise control over the expenditure of a dedicated research or engineering thread.

### Neural Vault
A secure, client-side encrypted storage for LLM API keys. It utilizes the browser's **SubtleCrypto API** over a **Secure Context** (HTTPS/localhost) to ensure keys remain private. Decryption is isolated in a dedicated **Web Worker**.

### Master Passphrase
The user-defined key used to encrypt and unlock the **Neural Vault**. This passphrase is never sent to the backend and exists only in volatile memory during an active session.

### Emergency Vault Reset
A terminal recovery protocol that purges all encrypted configurations from local storage. Used to regain entry to the system if a **Master Passphrase** is lost or if vault data becomes corrupted.

### Skill Synchronization (Sync)
The process of ensuring an agent's assigned **Skills** and **Workflows** are loaded into their `RunContext`. Originally hardcoded, these are now natively resolved from the dynamic file-system registry on every turn, allowing rapid iteration without restarts.

### Dynamic Skills
Custom, user-defined tools created via the Operative interface. They consist of a JSON schema defining parameters and an execution script (Bash, PowerShell, Python, etc.) that the engine securely runs as a subprocess.

### Passive Workflows
Markdown-based directives stored in the skills registry. They function as persistent "knowledge drops" or multi-step SOPs that are injected directly into an agent's system prompt during execution.

### Deterministic Workflows
Structured multi-step missions defined in the `workflows` table. Unlike passive workflows, these are active execution pipelines managed by the `WorkflowEngine`.

### Mission Cluster (Logical Cluster)
The frontend organizational unit for a group of agents. Clusters are managed via the `workspaceStore.ts` and persisted in the browser's LocalStorage. They determine the high-level objective and provide the "Strategic context" for the agents within.

### Workspace (Physical Sandbox)
The backend directory (located in `workspaces/`) used as a secure, isolated environment for an agent's file-based tool calls. Every Mission Cluster maps to a specific workspace to ensure agents can only interact with files relevant to their mission.

### File-System Swarm Vaults
The `/data/swarm_config/` directory where downloaded swarm templates are natively extracted and hot-loaded by the Rust engine. This file-system-first approach replaces static database configurations for declarative, version-controllable swarm installations.

### Swarm Template Ecosystem
A decentralized distribution model allowing users to discover and install pre-configured, industry-specific agent swarms (e.g., Legal, Healthcare) directly from the Tadpole OS dashboard.

### GitHub Native Hub
The native integration that allows the Tadpole OS Template Store to securely browse, search, and download swarm templates directly from a public GitHub repository without requiring a custom API client.

### Sapphire Shield
A rigorous zero-trust security protocol. In the context of the Template Ecosystem, it explicitly scans downloaded swarm configurations and refuses to initialize them if they request dangerous skills (e.g., `shell:execute`, `budget:spend`) until a human Overlord provides manual authorization.

### MCP (Model Context Protocol)
A standardized, open protocol used by Tadpole OS to unify tool discovery and execution. It allows the engine to treat internal functions, local scripts, and remote services as first-class citizens in the agent reasoning loop.

### System Delegate
A specialized type of MCP tool that, when invoked, triggers internal engine logic rather than an external process. Example: `recruit_specialist` delegates recruitment to the `AgentRunner`, allowing for parent-to-child strategic context injection.

### Atomic Synchronization (Load-then-Swap)
A fail-safe mechanism for the skills registry. The engine loads and validates new skills/workflows in a background buffer before hot-swapping the active memory registry, preventing configuration corruption or downtime during rapid updates.

### Process Guard (Execution Timeout)
An asynchronous watchdog that monitors dynamic skill execution. It enforces a strict **60-second limit** by default, automatically terminating any script that hangs or exceeds its resource window to maintain engine responsiveness.

### Lifecycle Hooks (SEC-04)
Security-first governance scripts (`pre-tool` and `post-tool`) executed by the engine before and after any tool call. They provide a governance-level auditing layer for sensitive operations.

### Mission Badge
A high-density UI component that displays the current operational objective. It features a rectangular design (`rounded-md`) and is **scrollable** to accommodate multi-step mission directives without impacting overall grid alignment.

### Benchmark Analytics Hub
The centralized system for recording and analyzing engine performance. It tracks latency (mean, p95, p99) and categorization for all mission-critical operations, allowing for historical comparisons and regression detection.

### Performance Delta
The calculated variance between two benchmark runs. Expressed as a percentage (e.g., -15% latency), it provides immediate feedback on whether a system update or model change has improved or degraded efficiency.

### Fiscal Burn (TPM)
The real-time tracking of aggregate token usage (Tokens Per Minute) across all active providers in the swarm, used to prevent exceeding budget allocations or provider quotas.

### Test Trace (Handshake)
A real-time connectivity diagnostic performed by the backend to verify that a provider configuration (**API Key**, **Network Endpoint**, and **Protocol**) is valid and reactive. In the UI, this is intercepted to ensure prerequisites are met before the handshake is initiated.

### Reactive Infrastructure
The architectural standard in **Tadpole OS** where the frontend state (**Zustand**) and communication layer (**TadpoleOSSocket**) are tightly coupled. Any change to engine settings triggers an immediate, non-disruptive reconnection of all system sensors.

### Capability Badge
A high-density UI component displayed on agent cards in the **Agent Manager**. These badges (represented by Code, File, and Terminal icons) indicate the count of assigned Skills, Workflows, and MCP Tools. Hovering over a badge reveals a detailed list of specific capabilities.

### Bulk Assignment (Skills Hub)
A high-efficiency operational feature in the **Skills & Workflows** view. It allows an operator to select a specific capability (Skill, Workflow, or MCP Tool) and sync it to multiple agent nodes simultaneously, bypassing individual configuration turns.

### Lazy Singleton Pattern
A performance-first initialization strategy used for the `TadpoleOSSocket`. The socket object is wrapped in a **Proxy** that only instantiates the real connection logic upon the first internal method call, preventing startup-time circular dependencies.

### Neural Map
An advanced SVG visualization layer within the Mission Cluster view. It uses real-time telemetry to draw animated "Connection Traces" between the Alpha Node and its Specialists, providing a high-fidelity mental model of the active swarm topology.

### Mission Analysis
The automated post-mission evaluation process. When enabled, the engine triggers a specialized auditing agent to review execution logs and provide performance insights.

### Success Auditor
The persona of the Mission Analysis Agent (ID 99). It is tasked with goal validation, tool efficiency auditing, and providing optimization prescriptions for future missions.

### Behavioral Drift
A semantic auditing mechanism within Mission Analysis that calculates the vector distance between an agent's final output and its core `IDENTITY.md` parameters. If the distance exceeds a configured threshold, the system flags the anomalous autonomy.

### Semantic Pruning
A token-reduction strategy used during extensive mission debriefs. Instead of analyzing raw logs entirely, the engine embeds them into a LanceDB vector space and extracts only the most critical pivot points (errors, decisions, blockers) to drastically reduce LLM context windows.

### Bunker Discovery
A defensive infrastructure protocol that enables the primary engine to scavenge the local network for secondary "Bunker" nodes. When a node is discovered, it is automatically integrated into the sovereign dashboard for cross-bunker mission orchestration.
### Bunker Cache
A high-performance, SQLite-backed storage system within the Rust engine that persists synthesized audio by text hash. It enables "Instant Replay" of common phrases, bypassing neural synthesis entirely for a zero-latency experience.

### Neural VAD (Silero)
Local Voice Activity Detection powered by the Silero ONNX model. It allows the engine to intelligently detect the start and end of user speech natively, enabling advanced segmentation and lowering cloud costs.

### Sovereign TTS (Piper)
A fast, local-first text-to-speech engine integrated via ONNX Runtime. It provides high-quality speech synthesis without external API dependencies, ensuring absolute privacy for agent responses.

### Sovereign STT (Whisper)
A local implementation of the Whisper neural model, allowing for high-fidelity speech-to-text transcription directly on the server host.

### PCM Streaming
The process of transmitting raw audio data (Pulse Code Modulation) in small 22kHz chunks over binary WebSockets. This enables real-time, low-latency playback where the user hears the agent start speaking before the full synthesis is complete.

### Detachable Tab Portal
A specialized operational interface that allows a dashboard sector (tab) to be detached into a native browser window. It utilizes a single-instance React Portal architecture to maintain a unified state and WebSocket connection across multiple monitors.

### Portal Style Sync
The automated process within the `PortalWindow` component that mirrors CSS variables, Tailwind classes, and theme metadata from the primary application to all detached windows, ensuring visual continuity.

### Single-Instance Detachment
An architecture where multiple browser windows share a single JavaScript heap and React tree. This eliminates the need for expensive cross-tab message passing and ensures that real-time data is synchronized across all views with zero latency.

### Multi-Tab Sovereign Interface
The high-performance navigation system in Tadpole OS that allows operators to maintain multiple active operational contexts (Missions, Hierarchy, Settings) in a single browser tab. Features a persistent **Multi-Tab Bar** located at the top of the viewport.

### Unified Tactical Header
The static, context-aware command surface at the top of every page. It dynamically swaps metrics and actions based on the active tab context.

### Sovereign Panel
A core design utility providing a standardized glassmorphic container for all principal page components, ensuring a cohesive "**Operations Center**" aesthetic across the entire codebase.

### Capability Import
The process of ingesting external `.md` or script-based skill and workflow definitions into the Tadpole OS registry. It utilizes a high-fidelity parser to convert documentation into structured agentic capabilities.

### Import Preview
A safety-first UI component that allows an operator to inspect the interpreted data and raw source of a capability before committing it to the system registry.

### Auto-Registration
A recursive discovery mechanism where the engine automatically captures and registers sub-agent capabilities. This ensures that intelligence discovered during a mission is permanently retained for future cross-swarm use.

### AI Services Category
A specialized registry namespace for capabilities discovered or generated by AI agents. This allows the system to distinguish between human-curated skills and those autonomously registered by the swarm.

### User Services Category
The default registry namespace for skills and workflows manually created or imported by a human operator.
